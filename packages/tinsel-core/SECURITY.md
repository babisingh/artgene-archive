# Security Policy — tinsel-core

## Reporting a Vulnerability

**Do not open a public GitHub issue for security vulnerabilities.**

Email the security team at **security@artgene.example** with:

- Description of the vulnerability and affected component
- Steps to reproduce (proof-of-concept code if applicable)
- Assessed impact (cryptographic break, key recovery, etc.)
- Your suggested severity level

We aim to acknowledge reports within **48 hours** and to release patches within
**14 days** for critical issues.  You will receive credit in the CHANGELOG
unless you prefer to remain anonymous.

## Security Design Notes

### 1. Key material

tinsel-core uses two independent secret keys:

| Key              | Purpose                          | Minimum size |
|------------------|----------------------------------|-------------|
| `spreading_key`  | HMAC-SHA3-256 spreading code     | 32 bytes    |
| `signing_key`    | HMAC-SHA3-256 authentication tag | 32 bytes    |

**These keys MUST be different.**  Reusing the same key for both purposes
collapses the independence between the embedding channel and the authentication
tag, enabling an adversary who knows the embedding pattern to forge signatures.

Derive both from a master secret using HKDF with distinct `info` labels:

```python
import hashlib, hmac
def hkdf_expand(prk: bytes, info: bytes, length: int) -> bytes:
    """HKDF-Expand (RFC 5869) over SHA-256."""
    t, okm, i = b"", b"", 0
    while len(okm) < length:
        i += 1
        t = hmac.new(prk, t + info + bytes([i]), hashlib.sha256).digest()
        okm += t
    return okm[:length]

master = os.urandom(64)   # store this in a secrets manager
spreading_key = hkdf_expand(master, b"tinsel-spreading-v1", 32)
signing_key   = hkdf_expand(master, b"tinsel-signing-v1",   32)
```

### 2. HMAC-SHA3-256 spreading

The spreading code is generated by counter-mode HMAC-SHA3-256:

```
block_i = HMAC-SHA3-256(spreading_key, label ‖ i.to_bytes(4, 'big'))
```

Security properties:
- **Pseudo-randomness**: without the key, the output is computationally
  indistinguishable from a uniform bit stream (assuming HMAC-SHA3-256 is a PRF).
- **Domain separation**: different `label` values (derived from `owner_id` and
  `timestamp_str`) produce independent sequences, preventing cross-registration
  correlation.

### 3. Reed-Solomon error correction

The embedded signature is protected by a systematic Reed-Solomon code over
GF(2^8) with primitive polynomial 0x11d.  RS parameters depend on tier:

| Tier     | RS(n,k)  | Correctable bytes |
|----------|----------|-------------------|
| FULL     | (32, 16) | 8                 |
| STANDARD | (16,  8) | 4                 |
| REDUCED  | ( 8,  4) | 2                 |
| MINIMAL  | ( 4,  2) | 1                 |
| DEMO     | none     | 0                 |

RS is applied **before** spreading, so the spread codeword has error correction
built in.

### 4. Known limitations

- **Short sequences (DEMO tier)**: sequences with fewer than 96 carrier bits
  use no RS protection.  A single codon mutation will corrupt the signature.
  DEMO-tier certificates should only be used for development and testing.

- **Codon randomisation attack**: an adversary who randomly shuffles synonymous
  codons will destroy the watermark.  The codon pattern is not authenticated
  (only the de-spread signature is).  Physical chain-of-custody evidence should
  supplement the watermark for high-stakes applications.

- **Side-channel via chi-squared**: a statistical test can confirm whether a
  DNA sequence was watermarked by TINSEL, but cannot recover the key material
  or forge a signature.

- **No forward secrecy**: compromising `spreading_key` allows an attacker to
  determine which sequences share the same key (but does not reveal the
  `signing_key` or allow forgery).

### 5. Test vectors

The keys `bytes.fromhex("aa" * 32)` and `bytes.fromhex("bb" * 32)` are used
**exclusively** in the unit test suite.  They are hardcoded to be obviously
synthetic (repeating single-byte patterns) and are safe to include in the
repository because they carry no secret value.

Any 32-byte key with non-trivial content must be stored in a secrets manager
(e.g. AWS Secrets Manager, HashiCorp Vault) and must never appear in source
code, test vectors, or log files.

## Threat Model

| Threat                                  | Mitigation                                   |
|-----------------------------------------|----------------------------------------------|
| Key recovery from codeword              | HMAC-SHA3-256 is a one-way PRF              |
| Signature forgery without signing_key   | HMAC-SHA3-256 preimage resistance           |
| Watermark stripping by codon shuffle    | RS + spreading increases required mutations  |
| Cross-registration correlation          | Per-registration labels in spreading code    |
| Side-channel timing (key comparison)    | Not applicable — pure Python, no secret-eq  |
| Quantum adversary (Grover)              | 128-bit FULL tier offers 64-bit post-quantum |
